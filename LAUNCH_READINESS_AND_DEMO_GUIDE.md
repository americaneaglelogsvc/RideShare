# Launch Readiness & Demo Guide

## 1. Platform Readiness Assessment

### 📱 Are the web assets responsive?
**Yes.** All web applications (Rider Web App, Admin Portal, Ops/Owner Consoles, and Public Website) are built using **React / Next.js** and styled with **Tailwind CSS**. Tailwind uses a mobile-first philosophy, utilizing responsive classes (like `md:`, `lg:`) to ensure interfaces look premium and scale gracefully across desktops, tablets, and mobile browsers.

### 🚗 Are mobile apps fully functional and ready for users?
They are **Requirement-Complete** and fully functional in a development/simulation environment, but they are **NOT ready for public users yet**. 
To get them into users' hands, you still need to:
1. Register **Apple Developer** and **Google Play** enterprise accounts.
2. Generate production certificates for Push Notifications (APNs/FCM) and Maps.
3. Build the standalone binaries (`.ipa` and `.aab`) using Expo Application Services (EAS Build).
4. Point the apps to a production Cloud URL (currently they point to your local development machine).

### ⚙️ Technical Debt & Missing Assets (GitHub Sync)
- **GitHub is 100% synchronized.** There is no pending code on your local machine.
- **Technical Debt / Next Steps**: 
  - We have comprehensive load tests (k6), chaos tests, and a Go/No-Go evaluation script, but we cannot run them until the backend is actually deployed to the cloud.
  - We need to swap out development API keys (Stripe, Twilio, Google Maps) for production keys.
  - A final WCAG accessibility and security vulnerability scan is required prior to public marketing.

### 🚀 Are we ready to deploy/launch for public view?
**Almost, but not today.** The codebase is feature-complete ("Dev Complete"), but the **infrastructure** is not live. 
The immediate next step is to trigger the GitHub Actions CI/CD pipeline to deploy the NestJS Gateway to **Google Cloud Run**, deploy the web apps to **Vercel/Netlify**, and run our intense load tests against the live URLs to ensure the servers don't crash under pressure.

## 2. Inventory of Local Demos & How to Trigger Them

Because the platform is not yet in the cloud, all demos must be run locally on your machine. 

### 🛑 Prerequisites for all demos
Before starting any demo, you must have the backend database and API running:
1. Open a terminal and start the database: `npx supabase start`
2. Open a second terminal, navigate to the gateway, and start the API:
   ```bash
   cd services/gateway
   npm run start:dev
   ```

### 🎯 Demo Flow 1: Rider Books a Ride
**Scenario:** A rider opens the generic web app, requests a ride, gets a price quote, and confirms.
**Trigger Instructions:**
1. Open a terminal and navigate to the Rider Web App:
   ```bash
   cd apps/rider-app
   npm run dev
   ```
2. Open your browser to `http://localhost:5173`.
3. Click "Book a Ride", enter a pickup and dropoff address (e.g., "JFK Airport" to "Times Square").
4. Observe the price calculation, select an economy vehicle, and confirm the booking.

### 🎯 Demo Flow 2: Driver Receives & Completes the Ride
**Scenario:** A driver goes online, receives the dispatch ping, picks up the rider, and completes the trip.
**Trigger Instructions:**
1. Open a terminal and navigate to the Driver Web App:
   ```bash
   cd apps/driver-app
   npm run dev
   ```
2. Open your browser to `http://localhost:5174` (or whatever port Vite assigns).
3. Log in as a driver and toggle your status to **Online**.
4. While the Rider is booking a ride in Demo 1, watch this screen. A full-screen dispatch offer will appear.
5. Accept the offer. Use the buttons to simulate the trip lifecycle: "Arrived at Pickup" -> "Start Trip" -> "Complete Trip".
6. Check the "Earnings" tab to see the ledger entry for the completed ride.

### 🎯 Demo Flow 3: Tenant Owner Console
**Scenario:** A local black-car business owner logs in to manage their fleet, drivers, and custom pricing policies.
**Trigger Instructions:**
1. Open a terminal and navigate to the Admin Portal:
   ```bash
   cd apps/admin-portal
   npm run dev -p 4400
   ```
2. Open your browser to `http://localhost:4400/owner`.
3. Navigate through the 16-module dashboard:
   - Go to **Fleet Management** to see registered vehicles.
   - Go to **Policy Center** to adjust the per-mile pricing markup for this specific tenant.
   - Go to **Reports** to see the aggregated revenue generated by Demo Flow 1 & 2.

### 🎯 Demo Flow 4: Native Mobile App Simulation
**Scenario:** Testing the native iOS/Android experience including offline-mode logic and biometrics (simulated).
**Trigger Instructions:**
1. Ensure your computer has the Expo Go app installed on your physical smartphone, or an iOS Simulator running.
2. Open a terminal and navigate to the Rider Native App:
   ```bash
   cd apps/rider-app-native
   npx expo start
   ```
3. Scan the QR code that appears in the terminal with your phone's camera (or press `i` to open in iOS simulator / `a` for Android).
4. The app will launch natively on your device. You can test the login, biometrics prompt, and booking interface via native UI components.
